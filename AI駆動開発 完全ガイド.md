1. 要件定義・設計の方法論
1.1 なぜ要件定義が重要か
AIコーディングエージェントは「仕様書通りに実装する」ことは得意ですが、「曖昧な指示から意図を汲み取る」ことは苦手です。
❌ 曖昧な指示 → AI が推測 → 意図と違う実装 → 手戻り
✅ 明確な仕様 → AI が正確に実装 → 一発で完成
💡 GitHubエンジニアリングチーム曰く: 「仕様書が真実の源泉となり、何が構築されるかを決定する」
1.2 インタビュー駆動アプローチ（推奨）
Claudeに質問させることで、漏れを防ぐ
## プロンプト例

以下のアイデアについて、要件定義を行いたいです。

アイデア: [簡単な説明]

私にインタビューしてください。以下の観点で質問を1つずつ行い、
私の回答を踏まえて次の質問をしてください：

1. ターゲットユーザーと課題
2. 競合・代替手段との差別化
3. 必須機能とNice-to-have
4. 成功の定義（KPI）
5. 技術的制約・予算・期限
6. エッジケースとリスク

**重要**: 当たり前の質問はせず、以下を引き出す質問をしてください：
- 隠れた前提
- 未検討のエッジケース
- 必要なトレードオフ

全ての質問が終わったら、PRD（Product Requirements Document）を作成してください。

1.8 デザイン制作フェーズ
要件定義の後、実装の前にデザイン制作フェーズを設けることで、手戻りを大幅に削減できます。
デザインフェーズの位置づけ
要件定義 → デザイン → 実装 → テスト
          ↑ここ

デザインなしで実装すると：
❌ 「なんか違う」→ 作り直し
❌ 一貫性のないUI
❌ ユーザー体験の考慮漏れ

デザインを挟むと：
✅ 実装前に見た目を確認できる
✅ デザインシステムで一貫性確保
✅ ユーザーフローの検証が可能

Pencilでの作業手順:
1. 新規ドキュメント作成
   File → New Document

2. ステンシルを選択（左パネル）
   - "Common Shapes" - 基本図形
   - "Android" - Androidアプリ用
   - "iOS" - iOSアプリ用  
   - "Sketchy GUI" - 手書き風（Lo-Fi向け）

3. 画面を作成
   - 部品をドラッグ&ドロップで配置
   - テキストをダブルクリックで編集

4. ページを追加
   Document → New Page

5. 画面間リンクを設定（オプション）
   ボタン選択 → プロパティ → Link To

6. エクスポート
   Document → Export Document → PNG/PDF

Mermaidコンポーネントの実装例:
'use client';

import React, { useEffect, useRef } from 'react';
import mermaid from 'mermaid';

interface MermaidProps {
  chart: string;
  className?: string;
}

export function Mermaid({ chart, className = '' }: MermaidProps) {
  const containerRef = useRef<HTMLDivElement>(null);

  useEffect(() => {
    mermaid.initialize({
      startOnLoad: true,
      theme: 'default',
      securityLevel: 'loose',
    });

    if (containerRef.current) {
      containerRef.current.innerHTML = chart;
      mermaid.run({ nodes: [containerRef.current] });
    }
  }, [chart]);

  return <div ref={containerRef} className={`mermaid ${className}`} />;
}

'use client';
import React, { useState } from 'react';

// 1. タブIDの型定義（全画面を列挙）
type TabId = 'home' | 'login' | 'signup' | 'detail' | 'mypage' | ...;

// 2. ナビゲーションprops（全コンポーネント共通）
interface NavigationProps {
  onNavigate: (tab: TabId) => void;
}

// 3. メインコンポーネント
export default function DesignPreviewPage() {
  const [activeTab, setActiveTab] = useState<TabId>('home');
  const tabs = [
    { id: 'home', label: 'ホーム' },
    { id: 'login', label: 'ログイン' },
  ];
  return (
    <div>
      {/* タブ切り替えUI */}
      {tabs.map((tab) => (
        <button key={tab.id} onClick={() => setActiveTab(tab.id as TabId)}>
          {tab.label}
        </button>
      ))}
      {/* 条件描画 */}
      {activeTab === 'home' && <HomePreview onNavigate={setActiveTab} />}
      {activeTab === 'login' && <LoginPreview onNavigate={setActiveTab} />}
    </div>
  );
}

// 4. 各画面コンポーネント
function HomePreview({ onNavigate }: NavigationProps) {
  return (
    <div className="min-h-screen bg-gray-50 pb-24">
      <Header />
      <main className="max-w-md mx-auto px-4 py-6">
        {/* コンテンツ */}
      </main>
    </div>
  );
}

デザインプレビューの完成度を高めるには、体系的なレビューが不可欠です。以下の観点を優先順位の高い順にチェックすることで、手戻りを最小限に抑えられます。
A. 構造・網羅性（最優先）
ページの過不足チェック
PRD・ユーザーフロー・ワイヤーフレームと突き合わせて、必要な画面が全て存在するか
ユーザーフローの各ステップに対応する画面があるか
エラー状態・空状態・ローディング状態の画面が考慮されているか
ユーザーフローの完結性
主要フロー（例：会員登録→プロフィール作成→イベント検索→参加申請→完了）が途切れずに辿れるか
分岐フロー（ログイン/未ログイン、主催者/参加者）が網羅されているか
異常系フロー（入力エラー、ネットワークエラー、権限不足）が考慮されているか
B. 画面遷移の整合性
到達不能画面の検出（grepベースの機械的チェックが有効）
# TabIdの定義を取得
grep "type TabId" page.tsx

# 各TabIdへの遷移回数を集計
grep -o "onNavigate('[^']*')" page.tsx | sort | uniq -c | sort -rn
この結果とTabIdの一覧を突き合わせれば、遷移が0回の画面（=到達不能）が即座に判明する。

各画面コンポーネントにonNavigateの呼び出しが存在するか確認
戻るボタン、ボトムナビ、いずれもない画面 = 行き止まり
特にモーダル・サブ画面（チャット会話、プロフィール閲覧など）は見落としやすい

横スクロールが不意に発生しないか

13.13 テスト戦略テンプレート
📝 普遍原則: この内容はClaude以外のAIツールでも適用可能
プロダクトの種類によって最適なテスト戦略が異なります。以下のテンプレートを参考に決定してください。
# テスト戦略ドキュメント

## 1. テストピラミッド構成

| レイヤー | 割合 | 対象 | ツール例 |
|---------|------|------|---------|
| Unit | 70% | 関数、クラス、モジュール | Jest, pytest, Vitest |
| Integration | 20% | API、DB境界、外部サービス連携 | Supertest, pytest + DB |
| E2E | 10% | ユーザーフロー全体 | Playwright, Cypress |

## 2. 各レイヤーの方針

### Unit テスト
- **対象**: ビジネスロジック、ユーティリティ関数、バリデーション
- **カバレッジ目標**: 80%以上（重要モジュールは90%以上）
- **モック方針**: 外部依存は全てモック化

### Integration テスト
- **対象**: 
  - DB操作（CRUD、トランザクション）
  - 外部API呼び出し
  - 認証・認可フロー
- **環境**: テスト用DB（Docker等）を使用
- **モック方針**: 外部APIのみモック、DBは実物

### E2E テスト
- **最小セット**（必須）:
  - [ ] ユーザー登録・ログイン・ログアウト
  - [ ] メインの価値提供フロー（例: 商品購入、投稿作成）
  - [ ] 課金・決済に関わるフロー（該当する場合）
  - [ ] データ作成・更新・削除の基本フロー
- **実行タイミング**: PR作成時、main/developマージ前

## 3. AIが書いたテストを疑う観点

- [ ] テストが **要件（PRD）を満たしているか**
- [ ] **境界条件** をカバーしているか（空、最大値、異常値）
- [ ] **偽陽性** がないか（常にパスするテスト）
- [ ] **偽陰性** がないか（本来失敗すべきケースがパスする）
- [ ] モックが **過剰** でないか（実際の挙動と乖離）
- [ ] テストの **可読性** は十分か

    - run: npm ci
    - run: npm run lint

13.16 セキュリティ最小基準
📝 普遍原則: この内容はClaude以外のAIツールでも適用可能
AI駆動開発で最低限守るべきセキュリティ基準です。
## 必須ルール（全プロジェクト共通）

### 1. シークレット管理
- [ ] APIキー、パスワードは **絶対にコードにハードコードしない**
- [ ] 環境変数または Secret Manager を使用
- [ ] .env ファイルは .gitignore に追加
- [ ] CI/CDのシークレットは暗号化して保存

### 2. 権限最小化
- [ ] DBユーザーは必要最小限の権限のみ付与
- [ ] APIキーはスコープを限定
- [ ] サービスアカウントは用途別に分離

### 3. 依存関係管理
- [ ] `npm audit` / `pip-audit` を定期実行
- [ ] Dependabotまたは同等のツールを有効化
- [ ] 重大な脆弱性は72時間以内に対応

### 4. ログ・監視
- [ ] 機密情報（パスワード、トークン、PII）は **ログに出力しない**
- [ ] エラーメッセージに内部情報を含めない
- [ ] 認証失敗、権限エラーはログに記録

### 5. 入力検証
- [ ] ユーザー入力は全てサニタイズ/バリデーション
- [ ] SQLインジェクション対策（パラメータ化クエリ）
- [ ] XSS対策（エスケープ処理）

### 6. 認証・認可
- [ ] パスワードはハッシュ化して保存（bcrypt等）
- [ ] セッション管理は信頼できるライブラリを使用
- [ ] CSRF対策を実装

> 📝 **普遍原則**: この内容はClaude以外のAIツールでも適用可能

```markdown
## 機能面
- [ ] **要件との整合**: PRD/タスク定義と一致しているか
- [ ] **過剰実装**: 要件にない機能を勝手に追加していないか
- [ ] **不足**: 要件を満たしていない部分がないか

- [ ] **ユーザーへのフィードバック**: エラー時にユーザーに何を表示するか

- [ ] **N+1問題**: ループ内でDBクエリを発行していないか
- [ ] **不要な再計算**: メモ化すべき処理がないか
- [ ] **大量データ**: ページネーション、ストリーミングが必要な箇所

- [ ] **コメント**: 複雑な処理に説明があるか
